# Changelog

All notable changes to noCluCal will be documented in this file. Format follows the conventional commits convention and groups changes under [Unreleased] / Added / Changed / Fixed / Removed.

## [Unreleased]

### Added

- feat(phase-2b): AES-256-GCM token encryption helpers at `src/lib/calendar/crypto.ts`. `encryptToken(plaintext)` and `decryptToken(ciphertext)` use Node's built-in `node:crypto`; no external dependencies. Ciphertext format is `v1:base64nonce:base64ciphertext` with the 16-byte GCM auth tag appended to the ciphertext blob. Each encryption uses a fresh random 12-byte nonce, so the same plaintext encrypts to a different output every time. The `v1:` version prefix enables future key rotation without a schema change.
- feat(phase-2b): lazy key loading from `process.env.TOKEN_ENCRYPTION_KEY` (base64-encoded 32 bytes). Module import has no side effects; the env var is read on first encrypt/decrypt call. Lazy loading is deliberate so Next.js's build-time module collection does not crash when the env var is unset.
- docs(phase-2b): `.env.example` documents `TOKEN_ENCRYPTION_KEY` with the `openssl rand -base64 32` generation command and the note that the value must decode to exactly 32 bytes. Real value lives in Bitwarden under "noClu Infrastructure", on the droplet's `/opt/noclucal/.env`, and on Robert's `.env.local`.
- docs(phase-2b): `CLAUDE.md` gains a `### Token encryption` subsection under `## Calendar abstraction layer` documenting algorithm, format, version-rotation strategy, key sourcing, lazy-load rationale, and the route-code contract on decryption failure.
- docs(phase-2b): `ROADMAP.md` restructured to reflect the four-step Phase 2 plan (2a / 2b / 2c / 2d). 2b's `Google provider implementation and token encryption helpers` is split into 2b (this PR, encryption helpers only) and 2c (Google provider implementation). What was previously labeled 2c (OAuth routes + UI) is now 2d.
- test(phase-2b): `tests/lib/calendar/crypto.test.ts` covers round trip (ASCII, empty string, unicode), nonce uniqueness across calls for the same plaintext, format shape, tamper detection in the ciphertext blob, the auth tag, and the nonce, unknown-version rejection, malformed-format rejection (too few and too many parts), missing-key error, wrong-length-key error, and cross-key decryption failure (proving the ciphertext is bound to its key).
- feat(phase-2a): `CalendarProvider` interface at `src/lib/calendar/types.ts`. Stateless contract: tokens are passed as arguments to every operation, never held on an instance. Methods cover the full OAuth lifecycle (`buildAuthorizationUrl`, `exchangeCode`, `refreshAccessToken`, `revoke`) plus calendar operations (`listCalendars`, `getFreeBusy`, `createEvent`, `updateEvent`, `deleteEvent`). Full JSDoc contract on every method. Webhook methods intentionally deferred to a future extension interface.
- feat(phase-2a): provider registry at `src/lib/calendar/providers/index.ts`. Exposes `registerProvider`, `getProvider` (throws if id not registered), `listProviders`, and `_resetRegistryForTests`. Eager registration model; Phase 2b ships a side-effecting `register-all.ts` that registers the Google provider at server startup.
- feat(phase-2a): `calendar_connections` schema at `src/lib/db/schema/calendar-connections.ts`. Columns: id uuid PK, user_id uuid FK (cascade delete), provider varchar(32), external_account_id text, external_account_email text, access_token_ciphertext text, refresh_token_ciphertext text, token_expires_at timestamptz, scopes text[], connected_at timestamptz default now(), last_synced_at timestamptz nullable. Unique index on (user_id, provider) enforces one-account-per-provider for MVP; defense-in-depth unique on (user_id, provider, external_account_id); lookup index on user_id.
- feat(phase-2a): first Phase 2 migration at `drizzle/migrations/0001_equal_guardsmen.sql`. Pure additive: `CREATE TABLE` plus three index statements. No extensions required. Applied locally via `pnpm db:migrate` and to CI via the existing job-level service container; will apply to `noclucal_prod` automatically on merge via the existing `migrate` Compose profile in `deploy.yml`.

### Fixed

- fix(dockerfile): moved the migrator stage to sit between `build` and `runner` so that `runner` remains the last (default) build target. The Phase 1c Dockerfile placed `migrator` last; docker-compose.yml's `web` service does not specify `target:`, so Docker defaulted to building the migrator stage for `web`. The web container ran `pnpm db:migrate:deploy`, exited 0 after migrations applied, and the restart policy relaunched it in an infinite loop. Migrations applied correctly (the table exists in `noclucal_prod`), but the Next.js runtime never started, so cal.noclulabs.com returned 502 from Caddy for ~20 minutes after the 1c deploy. The migrate Compose profile uses `target: migrator` explicitly, so it is unaffected by the reorder.
- ops(phase-1a-followup): Caddy access log block for cal.noclulabs.com removed from `/etc/caddy/Caddyfile` on the droplet because `/var/log/caddy/` is not writable by the Caddy user. Logged in ROADMAP as a deferred polish item; re-enable later by pre-creating the log file with the correct ownership.
- chore(phase-1a): rename `gitignore` (committed without the leading dot in Phase 0) to `.gitignore` so git actually honors the ignore rules.
