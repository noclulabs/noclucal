- feat(phase-4d): booking write flow. Operation order, load-bearing: (1) validate the invitee input with Zod; (2) `resolvePublicEventType` (unknown or disabled returns `not_bookable`); (3) live re-check via `getAvailableSlots` over a narrow window covering the slot, asserting an available slot with the same start instant still exists (else `unavailable`); (4) claim the slot with `createBooking`, where the `bookings_no_overlap_per_host` exclusion constraint is the authoritative taken moment (a `23P01` becomes the `conflict` path and creates no Google event); (5) best-effort Google event AFTER the claim, then `updateBookingGoogleRefs` stores the event id, html link, and Meet link. - feat(phase-4a): migration `drizzle/migrations/0003_hard_alex_power.sql`. Hand-edited to prepend `CREATE EXTENSION IF NOT EXISTS btree_gist;` (Drizzle does not auto-generate extension creation) and to add the `bookings_no_overlap_per_host` exclusion constraint after the `CREATE TABLE`: `EXCLUDE USING gist (host_user_id WITH =, tstzrange(starts_at, ends_at) WITH &&) WHERE (status = 'confirmed')`. ### Fixed

- fix(callback): post-callback redirects landed on `http://0.0.0.0:3000/settings/calendars` in production because the callback route derived the redirect origin from `new URL(request.url).origin`, which inside the Docker container behind Caddy resolves to the internal bind address rather than the public host. Replaced all six `url.origin` usages (one success redirect plus five error redirects) with a new `getAppOrigin()` helper that reads `AUTH_URL` and strips trailing slashes. New helper at `src/lib/app-url.ts`; six-case Vitest suite at `tests/lib/app-url.test.ts` covers the typical HTTPS and HTTP cases, single and multiple trailing-slash stripping, and the localhost fallback for both unset and empty-string `AUTH_URL`. OAuth tokens, encryption, and the DB row write path are unaffected by this bug; the row was being written correctly in every successful connect even though the user landed on a broken URL afterward.
- fix(dockerfile): moved the migrator stage to sit between `build` and `runner` so that `runner` remains the last (default) build target. The Phase 1c Dockerfile placed `migrator` last; docker-compose.yml's `web` service does not specify `target:`, so Docker defaulted to building the migrator stage for `web`. The web container ran `pnpm db:migrate:deploy`, exited 0 after migrations applied, and the restart policy relaunched it in an infinite loop. Migrations applied correctly (the table exists in `noclucal_prod`), but the Next.js runtime never started, so cal.noclulabs.com returned 502 from Caddy for ~20 minutes after the 1c deploy. The migrate Compose profile uses `target: migrator` explicitly, so it is unaffected by the reorder.
- ops(phase-1a-followup): Caddy access log block for cal.noclulabs.com removed from `/etc/caddy/Caddyfile` on the droplet because `/var/log/caddy/` is not writable by the Caddy user. Logged in ROADMAP as a deferred polish item; re-enable later by pre-creating the log file with the correct ownership.
